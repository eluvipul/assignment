KEY DIFFERENCES OBSERVED:

Without Index

Full Table Scan: The plan shows that the database reads every row in the movies table because there’s no supporting index.

High Cost: The optimizer assigns a relatively high cost since it must scan all rows sequentially.

Access Path: "TABLE ACCESS FULL" is the typical operation listed.

With Index

Index Range Scan: After creating an index on imdb_id, the plan shows "INDEX RANGE SCAN" or "INDEX FULL SCAN" depending on how the query is written.

Lower Cost: The optimizer cost drops significantly because the database can jump directly to the indexed values instead of scanning the entire table.

Reduced I/O: The number of blocks read decreases, improving performance especially for large tables.
